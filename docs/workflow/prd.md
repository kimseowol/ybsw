# PRD 프로세스 절차서

기획 ↔ PRD ↔ 코드를 연결하는 세 절차의 단일 원본. 스킬(`.agents/skills/prd/`)과 훅이 이 문서를 가리킨다.

## 공통 원칙
- PRD는 why 전용이다. 수치·알고리즘·구현 방법은 적지 않는다 (규칙 원본: AGENTS.md).
- 경계에는 의도가 깨지는 조건만 적는다. 요구사항 나열이 아니다.
- 결정 기록은 append-only — 경량 ADR 형식 차용. 결정이 뒤집히면 새 항목으로 대체를 선언하고 옛 항목을 링크한다.
- 코드 태그는 파일당 1개다 (주 소유 PRD). 여러 PRD가 소비하는 코드에 구현과 독립된 고유 why가 있으면 태그를 늘리지 말고 독립 PRD로 분리한다. 단순 추출·배선 코드는 기존 의도를 소유한 PRD 하나를 태그한다. 태그는 소유권이지 변경 귀속이 아니다 — 교차 작업의 결정 기록은 변경 이유가 속한 PRD에 적는다.
- 미확인 연결은 비워 두지 않고 라벨을 쓴다: `미구현` / `코드 미확인` / `기획 확인 필요`. 이 세 개 외의 어휘를 만들지 않는다. 라벨 없는 빈칸은 AI가 유사도로 메우는 지점이다.
- `docs/design/`은 수정 금지. 기획 문제는 "기획자 확인 필요"로 보고한다.

## 0. 작업 유형 판정

PRD를 찾거나 만들기 전에 모든 요청을 먼저 분류한다.

1. 요청 결과가 기능 의도, 지켜야 할 경계 또는 외부에서 관찰되는 동작을 바꾸는가?
   - **예**: 기능 작업이다. 기존 PRD가 그 why를 소유하면 2절로, 기존 PRD로 설명할 수 없는 독립 why가 있으면 1절로 간다.
   - **아니오**: 동작 불변 리팩토링·주석·포맷이다. 아래 "동작 불변 변경" 절차를 따른다.
2. 새 PRD 후보는 다음 세 질문을 모두 통과해야 한다.
   - 구현 클래스·파일·알고리즘이 바뀌어도 남는 의도인가?
   - 기존 PRD의 의도와 경계로 완전히 설명할 수 없는가?
   - 코드 배치나 에이전트 편의가 아니라 플레이어·콘텐츠 제작·운영 또는 공유 시스템에 독립적으로 보장할 계약인가?

`여러 PRD를 건드린다`, `새 공유 클래스를 만든다`, `충돌을 줄인다`, `에이전트가 찾기 쉽다`는 사실만으로는 새 PRD의 근거가 아니다. 이는 how이며, 기존 why를 유지하는 구조 변경이면 코드와 커밋 메시지가 원본이다.

판정 예시:

- Bootstrap을 동작 그대로 분할한다 → 동작 불변 리팩토링, 새 PRD 없음, `[prd-skip]`.
- 기존 상태 수명·죽음 순서 계약을 단일 API가 강제하도록 재배치한다 → PRD-002/003을 회귀 제약으로 읽는 동작 불변 리팩토링.
- 여러 기능이 공유할 새로운 전투 규칙과 외부 동작을 도입한다 → 기존 PRD가 소유하지 못할 때만 독립 PRD 후보.

## 1. PRD 생성

입력: 기획 문서, 피그마 캡처·복붙 텍스트, 구두 설명, 기존 코드(역산) — 무엇이든.

1. 입력물을 읽는다. GD 문서가 있으면 `source`에 ID를, 없으면 출처를 적는다 (예: `코드 역산`, `구두 설명 (날짜)`).
2. `docs/INDEX.md`에서 범위가 겹치는 PRD가 있는지 확인한다. 기존 PRD가 why를 소유할 수 있으면 새 PRD를 만들지 않고 기존 PRD 갱신을 제안한다.
3. 입력물에서 0절의 세 질문을 통과하는 독립 의도를 찾는다. 없으면 지어내지 말고 사용자에게 묻는다. 추정이면 `(추정)` 표시.
4. 아래 템플릿으로 `docs/prd/PRD-###-이름.md`를 `status: 초안`으로 작성한다. ID는 INDEX.md 마지막 번호 +1.
5. `docs/INDEX.md`에 행을 추가한다.
6. 사용자 승인 후 `status: 승인`. **승인 전 구현 금지.**

### 템플릿
```markdown
---
id: PRD-###
source: [GD-### | 코드 역산 | 구두 설명 (날짜)]
status: 초안
code: []
updated: YYYY-MM-DD
---
# 기능 이름

## 의도
이 기능이 왜 존재하는가. 플레이어·콘텐츠 제작·운영 또는 공유 시스템 관점에서 무엇을 보장해야 하는가. 3~5문장.

## 경계
- 지켜야 할 것: 의도를 깨뜨리면 안 되는 조건. 수치 아님.
- 하지 않을 것: 범위 밖. AI의 과잉 구현 방지.

## 결정 기록 (append-only)
- YYYY-MM-DD (출처): 결정 — 이유. 기각한 대안이 있으면 무엇을 왜.

## 열린 질문
- [ ] 기획자/개발자 확인: ...
```

## 2. PRD 구현 (코드 작업 전반)

1. 대상 PRD를 읽는다. `status: 승인`이 아니면 중단하고 사용자에게 알린다. PRD가 없으면 위 "PRD 생성"부터.
2. `source`의 기획 문서가 있으면 함께 읽는다.
3. `status: 구현중`으로 바꾸고 구현한다. 새·수정 파일 상단에 `// PRD-###` 태그 — 파일당 1개 (공통 원칙).
4. **결정 기록 실시간 append** — 이 절차의 핵심:
   - 대화 중 사용자가 방향을 정하면 그 자리에서 결정 기록에 추가한다. 출처는 `(사용자 피드백)`.
   - AI가 스스로 내린 유의미한 구현 결정도 추가한다. 출처는 `(AI 판단)`. 나중에 사용자가 뒤집을 수 있게 근거를 남긴다.
   - 형식: `- 날짜 (출처): 결정 — 이유. 기각한 대안: 무엇을, 왜.`
   - 작업이 끝난 뒤 몰아 쓰지 않는다 — 그때는 왜 그랬는지 이미 잊는다.
5. 기획 의도가 문제로 보이면 코드로 우회하지 말고 열린 질문에 올리고 보고한다.
6. 마무리: `code` 목록·`updated` 갱신, `status` 갱신, INDEX.md 갱신.

### 동작 불변 리팩토링·주석·포맷 (PRD 유무와 무관)
- 수정 파일 상단 태그의 PRD를 먼저 읽고 의도·경계를 회귀 제약으로 사용한다.
- 새 파일은 그 구조가 실현하는 기존 의도를 기준으로 주 소유 PRD 하나를 태그한다. 여러 호출자가 생긴다는 이유로 새 PRD를 만들지 않는다.
- 외부 동작과 PRD 의도·경계가 그대로면 PRD 결정 기록·상태·INDEX를 갱신하지 않는다. 구조 변경의 이유는 코드와 커밋 메시지가 원본이다.
- 커밋 메시지에 `[prd-skip]`과 구체적인 동작 불변 사유를 넣어 커밋 게이트를 통과한다.
- 작업 중 외부 동작이나 의도·경계를 바꿔야 함이 드러나면 즉시 중단하고 0절에서 기능 작업으로 다시 분류한다.

## 3. 동기화 검사 (드리프트 감사)

큰 병합 후나 주기 점검 시 실행. 자동 게이트가 못 잡는 의미 수준 드리프트를 잡는 최종 방어선이다.

드리프트 기준 (수치 변경은 드리프트가 아니다):
- 코드가 PRD의 경계를 위반
- 코드의 방향이 의도와 모순
- 결정 기록에 없는 유의미한 방향 전환이 코드에 존재
- 기획 문서(GD)의 `updated`가 PRD보다 최신인데 반영 안 됨
- `[prd-skip]`으로 통과한 커밋 중 실제로는 동작이 바뀐 것

절차:
1. `scripts/check-prd.ps1`을 실행한다 — 기계 검사(태그 규약, PRD 실존, `code` 경로 실존)와 `[prd-skip]` 커밋 목록이 나온다. 위반은 그대로 리포트에 싣는다.
2. INDEX.md의 각 PRD: 코드를 읽고 위 드리프트 기준으로 대조. 발견 시 `status: 드리프트` + 차이 기록.
3. 1의 `[prd-skip]` 목록에서 실제로는 동작이 바뀐 커밋이 있는지 diff로 확인 (남용 점검).
4. 리포트만 출력. 해소는 사용자 결정: 코드가 옳으면 결정 기록 추가 후 PRD 갱신, PRD가 옳으면 코드를 되돌린다.
