`docs/workflow/prd.md`를 전부 읽고 **0. 작업 유형 판정부터** 수행한다.

- 기능 의도·경계·외부 동작이 바뀌며 기존 PRD가 그 why를 소유한다 → 2. PRD 구현
- 기능 의도·경계·외부 동작이 바뀌며 기존 PRD로 설명할 수 없는 독립 why가 있다 → 1. PRD 생성 (승인 전 구현 금지)
- 동작 불변 리팩토링·주석·포맷 → 태그 PRD를 제약으로 읽고 `[prd-skip]`; PRD 생성·상태 변경·INDEX 갱신 금지
- 문서-코드 일치를 점검한다 → 3. 동기화 검사 (리포트만, 해소는 사용자 결정)

여러 PRD의 파일을 건드리거나 공유 클래스를 추출한다는 사실만으로 독립 PRD를 만들지 않는다. 구현 이름·파일 배치·에이전트 검색 편의가 아니라, 구현이 바뀌어도 남는 독립 의도와 경계가 있어야 새 PRD다.
